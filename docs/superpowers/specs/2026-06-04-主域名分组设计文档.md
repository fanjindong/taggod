# 主域名分组设计文档

## 目标

将标签分组从完整域名调整为主域名。调整后，`mail.google.com`、`docs.google.com` 和 `www.google.com` 会归入 `google.com` 分组，减少同一网站多个子域名带来的分组碎片。

## 范围

- 一键整理标签页时按主域名排序和创建 Chrome 原生分组。
- 只有同一主域名下存在两个及以上非固定标签时才创建 Chrome 原生分组，单个标签不创建分组，原因是单页分组会增加标签栏噪音。
- 同一主域名分组内部继续按完整子域名排序，原因是子域名通常代表不同产品或工作区，例如 `docs.google.com` 和 `mail.google.com` 应该稳定排列。
- 弹窗中的分组列表、搜索标签展示和域名数量统计使用同一主域名标识。
- 星标优先分组继续使用 `groupKey`，但新计算出的 `groupKey` 会从完整域名变为主域名。
- 保存会话时保存主域名分组；恢复旧会话时优先沿用已保存的 `groupKey`，新会话自然使用主域名。

## 实现策略

继续复用 `background.js` 中的 `getDomainKey(url)` 作为唯一分组入口，避免多个模块各自解析域名导致行为不一致。

域名归并规则：

1. 不能解析的网址归入 `其他`，原因是浏览器内部页面或异常网址不一定存在标准主机名。
2. `localhost`、内网短主机名、IPv4 和 IPv6 地址保持原值，原因是它们没有可注册主域名。
3. 普通域名取最后两段，例如 `a.b.example.com` 归入 `example.com`。
4. 常见多级公共后缀取最后三段，例如 `a.example.com.cn` 归入 `example.com.cn`，避免错误归入 `com.cn`。

排序规则：

1. 固定标签仍然排在最前面。
2. 主域名分组仍然遵守星标优先和当前标签栏顺序。
3. 主域名相同时，按完整主机名排序。
4. 完整主机名也相同时，保持原始左右顺序。

## 边界说明

本次不引入完整公共后缀列表，原因是插件无构建步骤且当前需求只需要覆盖常见中文和国际域名后缀。后续如果需要严格支持所有公共后缀，可以增加构建期生成的后缀表或引入轻量域名解析库。

## 验证方式

- 使用校验脚本验证常见子域名、多级公共后缀、内网主机、IP 地址和异常网址。
- 使用 `node --check background.js` 验证后台脚本语法。
- 使用 `node scripts/校验插件.js` 验证插件关键文件和域名归并行为。
