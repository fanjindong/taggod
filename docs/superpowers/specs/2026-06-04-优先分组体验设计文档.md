# 优先分组体验设计文档

## 背景与目标

当前 `标签之神` 的一键整理规则是固定标签优先、其余标签按域名排序并创建 Chrome 原生标签分组。这个规则稳定，但无法表达用户对某些工作分组的长期偏好。

本次增量目标是：用户不需要手写域名列表，也能让几个重要分组在每次整理后稳定排在前面。

## 方案取舍

推荐方案是 `星标分组 + 自动识别当前顺序`。

- 星标分组：弹窗展示当前窗口的域名分组，用户点击星标后，该分组以后整理时排在普通分组前面。
- 自动识别当前顺序：用户在 Chrome 标签栏里手动拖动分组后，插件整理时直接读取当前从左到右的分组顺序，不要求用户额外保存。

备选方案一是 `当前活跃分组自动靠前`。这个方案不需要配置，但排序会随着当前页面变化，稳定性不足，适合作为后续增强。

备选方案二是 `手写域名列表`。这个方案实现简单，但用户体验差，容易输错域名，也不适合频繁调整。

## 首版范围

本次增量包含以下能力：

- 在弹窗中增加“优先分组”区域。
- 展示当前窗口按域名聚合后的分组列表。
- 支持点击星标，让分组每次整理时排在普通分组前面。
- 支持取消星标，让分组回到普通排序。
- 一键整理会自动识别整理前当前窗口的分组顺序，避免用户手动拖到前面的分组被重新排到后面。

本次增量不包含以下能力：

- 手动输入域名。
- 拖拽调整弹窗内分组顺序。
- 自动学习用户长期行为。
- 跨设备同步优先分组。

## 数据结构

优先分组配置保存到 `chrome.storage.local` 的用户配置中。

```js
{
  priorityGroups: [
    {
      groupKey: "github.com",
      title: "github.com",
      starredAt: 1780540200000
    }
  ]
}
```

字段含义：

- `groupKey`：用于排序和匹配的域名标识。
- `title`：弹窗展示名称，首版与域名一致。
- `starredAt`：星标时间，用于后续排查和扩展，不直接参与首版排序。

## 整理规则

一键整理的新排序规则如下：

1. 固定标签仍然排在最前面。
2. 星标分组排在非星标分组前面。
3. 多个星标分组按整理前当前窗口从左到右的分组首次出现顺序排列。
4. 没有星标的分组，也按整理前当前窗口从左到右的分组首次出现顺序排列。
5. 如果无法得到当前顺序，再按域名字母顺序兜底。
6. 同一个域名内保持原始左右顺序。
7. 无法解析的网址归入 `其他`，没有优先配置时排在普通分组中。

固定标签继续保持最高优先级，是因为固定标签通常代表用户长期入口，不应该被普通整理规则挤到后面。

自动识别当前顺序是为了尊重用户刚刚在 Chrome 标签栏里完成的手动拖拽结果。星标只决定哪些分组进入优先区，不决定多个星标分组之间的顺序。

## 弹窗交互

弹窗增加“优先分组”区域，位置放在常用操作按钮下方、搜索框上方。

区域包含：

- 当前窗口分组列表。
- 每个分组一枚星标按钮。
- 分组名称和标签数量。

用户点击星标后：

1. 弹窗发送 `toggle-priority-group` 消息给后台。
2. 后台更新 `settings.priorityGroups`。
3. 弹窗刷新分组列表。
4. 状态提示显示“已置顶分组”或“已取消置顶”。

## 后台接口

新增消息：

- `toggle-priority-group`：切换某个分组是否优先。

`get-state` 返回新增字段：

- `groups`：当前窗口分组列表。
- `settings.priorityGroups`：已保存的优先分组配置。

## 异常处理

- 当前窗口没有可识别分组时，显示空状态。
- 用户星标 `其他` 分组时允许保存，因为浏览器内部页面也可能是用户常用入口。
- 保存失败时显示失败提示，不改变当前弹窗状态。

## 测试策略

- 验证弹窗可以展示当前窗口分组列表和标签数量。
- 验证点击星标后，刷新弹窗仍能看到星标状态。
- 验证一键整理后，星标分组排在普通分组前，多个星标分组按当前标签栏顺序排列。
- 验证取消星标后，该分组回到普通域名排序。
- 验证 `node scripts/校验插件.js` 通过。

- 已明确新增存储字段和后台消息。
