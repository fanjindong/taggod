# 可配置分组阈值设计文档

## 目标

当前一键整理和工作集恢复时，只有同一主域名下存在两个及以上非固定标签才创建 Chrome 原生标签分组。这个阈值写死在后台逻辑里，用户无法按自己的标签使用习惯调整。

本次目标是把“创建原生分组所需的最少标签数”做成用户配置。默认值保持 `2`，原因是现有行为已经避免了单标签分组带来的标签栏噪音，老用户升级后不应该感知到默认行为变化。

## 范围

- 在用户配置中新增 `minTabsPerGroup` 字段，表示同一主域名下至少有多少个非固定标签才创建 Chrome 原生分组。
- 弹窗新增一个轻量设置入口，允许用户输入正整数阈值。
- 用户保存阈值后，立即对当前窗口已有标签重新梳理，原因是旧分组如果继续保留，会让界面状态和新配置不一致。
- 一键整理当前窗口时读取该阈值。
- 恢复工作集并重建分组时读取同一阈值，原因是恢复行为应该和用户当前整理规则一致。
- 弹窗的“优先分组”列表只展示当前阈值下会创建 Chrome 原生分组的主域名，原因是单标签或数量不足的主域名没有实际分组，不需要提供星标入口。

## 不做范围

- 不新增独立设置页，原因是当前插件没有设置页，单个数字配置放在弹窗内更直接。
- 不支持按主域名分别设置不同阈值，原因是本次需求只要求把全局硬编码阈值改成用户可配置。
- 不改变固定标签规则，固定标签仍不参与原生分组创建，原因是 Chrome 固定标签区域本身不适合被批量移动进普通分组。

## 配置策略

`DEFAULT_SETTINGS` 新增：

```js
minTabsPerGroup: 2
```

`normalizeSettings(settings)` 负责兜底：

1. 如果旧配置没有该字段，使用默认值 `2`。
2. 如果字段不是有效正整数，回退到 `2`。
3. 如果用户输入小于 `1`，回退到 `2`，原因是 `0` 或负数会让配置语义不清晰。

## 交互策略

弹窗新增“分组阈值”设置行，展示在“优先分组”区域上方或区域标题附近。

设置行包含：

- 数字输入框，最小值为 `1`。
- 保存按钮或输入变化后的显式保存动作。
- 当前状态提示复用底部状态文本，例如“已保存分组阈值：至少 3 个标签”。

推荐使用显式保存按钮，原因是 Chrome 插件弹窗会频繁关闭，显式保存能让用户清楚知道配置已经写入本地存储。

保存成功后立即触发当前窗口重新梳理：

1. 如果某个已有 Chrome 原生分组对应的主域名标签数量小于新阈值，则取消该原生分组。
2. 如果某个主域名标签数量大于等于新阈值，则创建或保留原生分组。
3. 固定标签仍然不参与分组判断，原因是固定标签区域和普通标签分组的交互方式不同。
4. 重新梳理只作用于当前窗口，原因是弹窗配置入口面向当前正在处理的标签现场，跨窗口自动改动会超出用户预期。

## 后台策略

新增一个小函数，例如 `shouldCreateNativeGroup(tabIds, settings)`：

```js
return tabIds.length >= settings.minTabsPerGroup;
```

一键整理和工作集恢复都通过这个函数判断是否调用 `chrome.tabs.group`。这样阈值判断只存在一个入口，避免后续两个流程行为不一致。

新增 `reconcileCurrentWindowGroups(settings)` 用于配置保存后的重新梳理。它负责查询当前窗口标签，按主域名聚合非固定标签，并按新阈值修正 Chrome 原生分组：

1. 对不满足阈值的已有分组调用 `chrome.tabs.ungroup(tabIds)`。
2. 对满足阈值的主域名调用 `chrome.tabs.group({ tabIds })` 并更新组名和颜色。
3. 数量不足的主域名保持未分组，原因是新阈值是用户刚刚确认的分组噪音控制规则；当用户把阈值设为 `1` 时，单个非固定标签也会创建为分组。

该函数不替代一键整理的排序能力。配置保存后的重新梳理只保证分组数量符合新阈值，不主动改变用户当前标签左右顺序，原因是用户可能只想调整分组显隐规则，而不是立刻重排整个窗口。

## 消息流

弹窗通过现有 `chrome.runtime.sendMessage` 发送新动作，例如 `update-settings`，只传需要更新的字段：

```js
{
  action: 'update-settings',
  settings: {
    minTabsPerGroup: 3
  }
}
```

后台读取现有设置，合并新字段，经过 `normalizeSettings` 后写回 `chrome.storage.local`。这样可以保留 `priorityGroups` 等旧设置，不会因为更新阈值误覆盖其他配置。

后台写入配置成功后调用 `reconcileCurrentWindowGroups(settings)`，并返回本次取消分组和创建分组的数量。弹窗状态提示可以显示“已保存分组阈值，已重新梳理当前窗口分组”。

## 验证方式

- 校验 `normalizeSettings({})` 会得到 `minTabsPerGroup: 2`。
- 校验无效值、负数、零、小数和字符串会回退到 `2`。
- 校验有效正整数会被保留。
- 校验一键整理和工作集恢复都使用同一个阈值判断函数。
- 校验调高阈值后，当前窗口里不满足新阈值的既有原生分组会被取消。
- 校验调低阈值后，当前窗口里满足新阈值的主域名会被创建为原生分组。
- 运行 `node --check background.js`、`node --check popup.js` 和 `node scripts/校验插件.js`。
