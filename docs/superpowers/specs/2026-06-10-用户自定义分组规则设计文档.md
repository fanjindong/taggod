# 用户自定义分组规则设计文档

## 背景

`标签之神` 当前已经支持按主域名自动分组、分组阈值、优先分组、短显示名和工作集恢复。默认主域名规则稳定、低配置，但无法覆盖用户真实工作里的跨站点场景。

典型例子包括：

- 一个项目同时散落在代码仓库、需求文档、设计稿、任务系统和线上环境里。
- 一个主题同时涉及多个 AI 工具、搜索结果、参考资料和内部后台。
- 同一个大站点里不同路径代表完全不同的项目或客户，不应该全部挤进同一个主域名分组。

因此需要给用户定义分组规则的能力，让用户可以用可理解的条件把标签归入指定分组。

## 目标

- 支持用户定义中等复杂度的分组规则。
- 规则既可以覆盖默认主域名分组名称，也可以把不同网站归入同一个自定义分组。
- 多条规则同时命中时，按规则列表从上到下匹配，第一条命中规则生效。
- 没有命中自定义规则的标签继续使用现有主域名分组。
- 自定义分组继续复用现有分组阈值、优先分组、排序、原生分组创建和工作集恢复流程。
- 弹窗提供轻量管理入口，支持新增、启用、停用、删除、上移和下移规则。

## 非目标

- 不支持用户编写脚本，原因是脚本规则会显著增加学习成本和安全风险。
- 首版不支持正则表达式，原因是大多数用户只需要“包含”或“等于”这类直观条件。
- 首版不支持复杂条件组，例如“条件一或条件二，并且条件三”，原因是弹窗空间有限，复杂表达式会让规则难以解释。
- 首版不做独立设置页，原因是当前插件的产品形态仍以弹窗轻量操作为主。
- 首版不做跨设备同步，原因是当前配置都保存在本地 `chrome.storage.local`。

## 方案取舍

### 方案一：只做分组重命名

该方案只允许用户把 `github.com` 显示为“代码仓库”等名称，内部仍按主域名归组。

优点是实现简单，风险低。缺点是无法把 GitHub、飞书文档和 Jira 同时归到“项目 A”，不满足跨站点自定义分组的核心目标。

### 方案二：轻量条件规则系统

该方案允许用户创建规则，每条规则包含规则名称、目标分组名、启用状态和若干匹配条件。标签命中规则后使用该规则指定的自定义分组。

优点是覆盖主要使用场景，同时保持规则模型可解释。缺点是需要增加规则管理界面和后台匹配逻辑。

本次推荐该方案。

### 方案三：高级规则系统

该方案支持正则表达式、复杂条件组、模板变量和导入导出。

优点是能力最强。缺点是配置门槛高，误配后难排查，也容易把插件从“一键整理工具”推向复杂规则编辑器。首版不采用。

## 规则模型

用户配置新增 `groupRules` 字段，保存在现有 `settings` 里：

```js
{
  groupRules: [
    {
      id: 'rule-1780540200000',
      name: '项目 A',
      enabled: true,
      targetGroupKey: 'custom:项目 A',
      targetTitle: '项目 A',
      conditions: [
        {
          field: 'hostname',
          operator: 'contains',
          value: 'github.com'
        },
        {
          field: 'path',
          operator: 'contains',
          value: '/my-org/project-a'
        }
      ],
      createdAt: 1780540200000,
      updatedAt: 1780540200000
    }
  ]
}
```

字段含义：

- `id`：规则唯一标识，用于编辑、删除和排序。
- `name`：规则展示名称，帮助用户理解这条规则的用途。
- `enabled`：是否启用规则，停用后保留配置但不参与匹配。
- `targetGroupKey`：命中后使用的内部分组键。覆盖主域名分组时可以使用真实主域名，例如 `github.com`；创建跨站点自定义分组时使用 `custom:` 前缀，例如 `custom:项目 A`，原因是它需要和真实主域名分组区分。
- `targetTitle`：命中后展示给用户的分组名。
- `conditions`：匹配条件列表，首版采用“全部满足”。
- `createdAt`：创建时间，用于排查和后续扩展。
- `updatedAt`：最近更新时间，用于后续展示或同步冲突处理。

## 条件字段

首版支持以下字段：

- `hostname`：完整域名，例如 `github.com`、`docs.google.com`。
- `primaryDomain`：主域名，例如 `google.com`。
- `path`：网址路径，例如 `/my-org/project-a/issues`。
- `url`：完整网址。
- `title`：标签标题。

这些字段覆盖了大多数可理解规则：用户既可以按站点归组，也可以按路径区分项目，还可以用标题匹配内部系统页面。

多条规则可以指向同一个 `targetGroupKey`。这是跨站点归组的主要方式，例如一条规则匹配 GitHub 项目仓库，另一条规则匹配 Jira 项目面板，两条规则都指向 `custom:项目 A`，最终会合并为同一个“项目 A”分组。

## 匹配操作

首版支持以下操作：

- `contains`：包含指定文本，适合路径、标题和网址。
- `equals`：完全等于指定文本，适合主域名和完整域名。
- `startsWith`：以指定文本开头，适合路径前缀。

所有匹配默认忽略大小写。原因是域名天然大小写不敏感，网页标题和路径里大小写差异通常也不应该影响用户规则命中。

空条件值不允许保存，原因是空字符串会导致大量标签被意外命中。空条件列表也不允许保存，原因是一条没有条件的规则会吞掉所有非固定标签。

## 分组计算流程

新增统一入口 `getResolvedGroupInfo(tab, settings)`，负责为单个标签计算最终分组信息。

流程如下：

1. 固定标签不参与自定义规则匹配，继续保留在固定标签区域，原因是固定标签通常代表用户长期入口。
2. 对非固定标签，按 `settings.groupRules` 的数组顺序从上到下检查。
3. 跳过未启用规则。
4. 一条规则的全部条件都满足时，立即返回该规则的 `targetGroupKey` 和 `targetTitle`。
5. 如果没有规则命中，回退到现有 `getDomainKey(url)` 主域名分组。
6. 后续排序、阈值判断、星标判断和原生分组创建都使用最终分组键。

该入口需要替代当前直接调用 `getDomainKey(url)` 的分组场景，避免整理、弹窗列表、工作集保存和工作集恢复出现不同规则。

## 排序规则

引入自定义规则后，整体排序继续沿用现有原则：

1. 固定标签排在最前。
2. 达到阈值的真实分组排在零散单标签前面。
3. 优先分组排在普通分组前面。
4. 同等级分组按整理前当前窗口从左到右的首次出现顺序排列。
5. 无法得到当前顺序时，按最终分组键排序兜底。
6. 同一最终分组内，优先按完整域名排序，再按原始标签位置兜底。

自定义分组不单独获得更高排序优先级。原因是用户已经可以通过“优先分组”表达置顶意图，规则系统只负责“归到哪里”，不额外负责“排到哪里”。

## 分组阈值关系

自定义规则命中的标签先归入自定义分组，再参与分组阈值判断。

例如阈值为 `2` 时：

- 一个 GitHub 标签命中“项目 A 仓库”规则，一个 Jira 标签命中“项目 A 任务”规则，两条规则都指向 `custom:项目 A`，则“项目 A”有两个标签，会创建 Chrome 原生分组。
- 只有一个标签命中“项目 A”，则该标签仍保持零散状态，原因是当前阈值规则要求至少两个标签才创建原生分组。

这样可以保证用户熟悉的噪音控制逻辑不被自定义规则绕过。

## 优先分组关系

优先分组继续基于最终分组键保存。

- 主域名分组仍保存类似 `github.com` 的键。
- 自定义分组保存类似 `custom:项目 A` 的键。

当用户修改规则的 `targetGroupKey` 时，旧优先配置不会自动迁移。首版编辑规则时尽量保持 `targetGroupKey` 稳定，只允许修改展示名 `targetTitle`。原因是自动迁移可能误改用户已有星标配置。

## 显示名关系

主域名分组继续使用现有短显示名逻辑。

自定义分组直接使用 `targetTitle` 作为显示名，不再经过公共后缀裁剪。原因是自定义名称本身就是用户输入的可读名称，如果继续裁剪反而可能破坏含义。

如果自定义分组标题和主域名短显示名冲突，展示层可以显示标题旁的完整分组键提示，例如 `项目 A（custom:项目 A）`。内部匹配仍以最终分组键为准，不会因为标题相同而合并错误分组。

## 弹窗交互

弹窗在“更多工具”中增加“分组规则”入口。

规则列表展示：

- 规则名称。
- 目标分组名。
- 条件摘要，例如“域名包含 github.com，路径包含 /project-a”。
- 启用状态。
- 上移、下移、编辑、删除按钮。

新增或编辑规则时使用轻量表单：

- 规则名称。
- 目标分组名。
- 条件列表。
- 每个条件包含字段、操作和值。

首版为了降低实现复杂度，可以限制每条规则最多 `5` 个条件。这个数量足够覆盖“域名 + 路径 + 标题”等常见组合，也能避免弹窗表单过长。

保存规则后立即刷新当前窗口分组预览和优先分组列表。是否立即执行一键整理由用户点击“整理当前窗口”决定，原因是保存规则属于配置动作，不应该隐式移动用户标签。

## 后台接口

新增或扩展以下消息：

- `create-group-rule`：新增规则。
- `update-group-rule`：更新规则名称、目标标题、条件和启用状态。
- `delete-group-rule`：删除规则。
- `move-group-rule`：调整规则顺序。
- `update-settings`：继续用于保存简单设置，但不直接承载复杂规则编辑，原因是规则编辑需要更细的校验和错误提示。

后台保存规则前必须做归一化：

1. 去除名称、目标标题和条件值两侧空白。
2. 校验名称和目标标题不能为空。
3. 校验字段和操作必须来自白名单。
4. 校验条件数量至少为 `1` 且最多为 `5`。
5. 新规则追加到列表末尾，原因是靠前规则优先级更高，追加末尾更不容易抢走旧规则命中。

## 工作集关系

工作集保存时可以继续保存每个标签当时的最终分组键，用于展示历史分组数量。

工作集恢复时推荐使用当前规则重新计算分组，而不是强制沿用保存时的旧规则结果。原因是用户修改分组规则后，通常期望恢复出来的标签也遵守最新整理规则。

如果后续需要“按保存时分组恢复”，可以在工作集里增加恢复选项，首版不做。

## 错误处理

- 规则名称为空时提示“规则名称不能为空”。
- 目标分组名为空时提示“目标分组名不能为空”。
- 条件值为空时提示“匹配内容不能为空”。
- 条件数量超过上限时提示“每条规则最多 5 个条件”。
- 删除规则前需要确认，原因是规则删除后无法从本地配置自动恢复。
- 保存失败时不更新前端本地状态，避免界面显示和真实配置不一致。

## 验证方式

- 校验没有规则时，现有主域名分组行为不变。
- 校验规则命中后，标签使用自定义分组键。
- 校验跨域名标签命中同一规则后进入同一分组。
- 校验多条规则同时命中时，列表中更靠前的规则生效。
- 校验停用规则不会参与匹配。
- 校验空名称、空目标分组名、空条件值和未知字段不能保存。
- 校验自定义分组仍遵守 `minTabsPerGroup`。
- 校验自定义分组可以设为优先分组。
- 校验工作集恢复时使用当前规则重新计算分组。
- 运行 `node --check background.js`、`node --check popup.js` 和 `node scripts/校验插件.js`。

## 设计自检

- 文档聚焦用户自定义分组规则，没有扩展到搜索、云同步或独立管理页。
- 规则模型明确了冲突处理方式：按列表顺序第一条命中。
- 自定义规则和现有主域名、阈值、星标、显示名、工作集关系均已说明。
- 已明确首版不支持脚本、正则和复杂条件组，避免实现范围失控。
- 错误处理和验证方式覆盖了主要边界条件。
