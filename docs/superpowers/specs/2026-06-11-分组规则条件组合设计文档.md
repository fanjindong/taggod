# 分组规则条件组合设计文档

## 背景

当前自定义分组规则使用简单条件数组，规则内多个条件固定为 AND 关系。这个模型不适合“多个域名进入同一个分组”的常见场景，因为用户需要创建多条规则并填写相同目标分组名，配置成本高，也容易因为规则顺序产生理解偏差。

本次调整把入口收敛为“创建分组规则”：一条规则对应一个唯一目标分组名，规则内部通过条件组合表达匹配范围。

## 目标

- 一条分组规则只对应一个目标分组名。
- 规则内支持条件组，用户可以选择“满足全部”或“满足任一”。
- 支持最多两层条件组，覆盖常见的“多个域名 OR，再叠加路径/标题限定”的场景。
- 不兼容旧的 `conditions` 数据结构，已有旧规则需要用户重新创建。
- 所有提示、文档和错误信息保持中文。

## 非目标

- 不支持无限嵌套，原因是插件弹窗空间有限，过深表达式会显著增加理解和编辑成本。
- 不做旧规则自动迁移，原因是用户已明确不需要兼容旧规则。
- 不引入外部规则引擎或表达式语言，避免增加插件体积和安全风险。

## 数据模型

每条规则使用 `conditionTree` 表达匹配逻辑：

```js
{
  id: 'rule-xxx',
  name: '项目 A',
  enabled: true,
  targetGroupKey: 'custom:项目 A',
  targetTitle: '项目 A',
  minTabsPerGroup: null,
  conditionTree: {
    type: 'group',
    logic: 'and',
    children: [
      {
        type: 'group',
        logic: 'or',
        children: [
          { type: 'condition', field: 'hostname', operator: 'contains', value: 'github.com' },
          { type: 'condition', field: 'hostname', operator: 'contains', value: 'docs.example.com' }
        ]
      },
      { type: 'condition', field: 'path', operator: 'contains', value: 'project-a' }
    ]
  },
  createdAt: 1,
  updatedAt: 1
}
```

示例含义：

```text
(域名包含 github.com 或 域名包含 docs.example.com) 并且 路径包含 project-a
```

## 规则限制

- 顶层必须是条件组。
- 条件组 `logic` 只允许 `and` 或 `or`。
- 条件节点只允许字段：域名、主域名、路径、网址、标题。
- 条件节点只允许操作：包含、等于、开头是。
- 条件值必须填写。
- 最多两层条件组：顶层组可以包含子组，子组只能包含条件，不能再包含子组。
- 单条规则最多保留 8 个条件节点。这个上限比旧版 5 个略高，是为了让“多个域名 OR”有足够空间，同时避免弹窗表单过长。

## 后台匹配

后台新增条件树归一化与匹配函数：

- `normalizeConditionNode(node, depth)`：归一化单个条件或条件组。
- `normalizeConditionTree(tree)`：归一化整棵条件树。
- `doesConditionTreeMatchTab(tree, tab)`：按条件树判断标签是否命中。

匹配规则：

- `and`：所有子节点都命中才算命中。
- `or`：任一子节点命中就算命中。
- 空组无效，归一化时直接丢弃。
- 无有效条件树的规则无效，不会进入设置。

## 弹窗交互

“分组规则”表单调整为条件组编辑器：

- 顶层显示逻辑选择：满足全部 / 满足任一。
- 顶层支持添加“条件”和“条件组”。
- 子条件组显示逻辑选择：满足全部 / 满足任一。
- 子条件组只支持添加“条件”，不支持继续添加条件组。
- 条件行仍包含字段、操作和值输入框。
- 删除最后一个条件时，如果导致条件组为空，保存时提示“至少需要一个匹配条件”。

推荐配置示例：

```text
目标分组名：项目 A
顶层：满足全部
子组：满足任一
  - 域名 包含 github.com
  - 域名 包含 docs.example.com
  - 域名 包含 jira.example.com
普通条件：
  - 路径 包含 project-a
```

## 错误处理

- 目标分组名为空：提示“目标分组名不能为空”。
- 条件值为空：提示“匹配内容不能为空”。
- 条件树为空：提示“至少需要一个匹配条件”。
- 超过两层条件组：提示“条件组最多支持两层”。
- 条件数量超过上限：提示“每条规则最多 8 个条件”。

错误提示在表单内用红色突出展示，同时同步到底部状态栏。

## 测试策略

- 校验脚本覆盖条件树归一化：
  - 有效 AND 条件组保留。
  - 有效 OR 条件组保留。
  - 空组、空值条件、非法字段和非法操作会被丢弃。
  - 超过两层的子组会被丢弃或报错，具体按归一化入口约定执行。
- 校验脚本覆盖匹配：
  - `and` 必须全部命中。
  - `or` 任一命中即可。
  - 两层组合表达式能正确匹配多个域名进入同一个目标分组。
- 校验脚本覆盖后台消息：
  - 创建规则必须使用 `conditionTree`。
  - 旧 `conditions` 不再作为有效规则入口。
- 校验脚本覆盖弹窗：
  - 存在条件组逻辑选择控件。
  - 存在添加条件组入口。
  - 错误提示红色样式存在。

## 自检

- 本设计已明确不兼容旧规则。
- 本设计没有引入无限嵌套，复杂度受控。
- 本设计覆盖了多个域名进入同一分组的目标场景。
- 本设计包含后台、弹窗、错误处理和测试要求。
