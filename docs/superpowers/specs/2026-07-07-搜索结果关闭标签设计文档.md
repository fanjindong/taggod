# 搜索结果关闭标签设计文档

## 背景

`标签之神` 当前搜索结果主要服务“找到并切换标签页”。搜索增强已经支持多关键词、缩写、最近关闭标签页和跨窗口结果，用户能更快定位目标页面。

当用户通过搜索发现某个已打开标签页已经不需要时，当前只能先切换过去，再手动关闭浏览器标签页。这个流程会打断用户正在整理搜索结果的上下文，也不适合连续清理多个无用页面。

本次迭代在搜索结果中增加单个标签关闭能力，让用户可以在不离开弹窗的情况下处理明确无用的已打开标签页。

## 目标

- 搜索结果中的已打开普通标签页可以被单独关闭。
- 默认点击结果仍然是跳转，不改变现有最高频路径。
- 关闭入口默认保持轻量，避免搜索列表变得拥挤。
- 关闭成功后立即从当前结果列表移除该标签页。
- 固定标签、当前聚焦窗口的当前活动标签和正在播放音频的标签默认不提供关闭入口，降低误关风险。
- 所有提示文案使用中文。

## 非目标

- 不做批量关闭，原因是批量关闭误操作成本更高，应该独立设计确认流程。
- 不支持关闭最近关闭结果，原因是它们不是已打开标签页。
- 不新增 `Delete` 或 `Backspace` 快捷键关闭，原因是搜索框聚焦时这类按键容易和文本编辑冲突。
- 不做关闭后撤销，原因是浏览器已有最近关闭恢复能力，本次先保持能力轻量。
- 不改变智能去重、工作集、一键整理和自定义分组规则逻辑。

## 方案比较

### 方案一：每条结果常驻关闭按钮

每条搜索结果右侧始终展示“关闭标签”按钮。

优点是入口最明显，用户不需要学习额外交互。缺点是搜索结果会更拥挤，尤其是标题和网址较长时，常驻按钮会压缩主要信息，也更容易被误点。

### 方案二：悬停、聚焦或键盘选中时展示关闭按钮

搜索结果默认保持现有视觉结构。用户鼠标悬停、结果项获得焦点或键盘选中某条结果时，该结果右侧展示关闭按钮。

优点是保持默认列表干净，同时在用户关注某条结果时提供操作入口。缺点是实现时需要同时处理鼠标、键盘选中和可访问性状态。

### 方案三：更多操作菜单中提供关闭标签

每条结果提供一个“更多操作”按钮，点击后在菜单里选择“关闭标签”。

优点是最安全，也方便后续扩展复制链接、移动到当前窗口等动作。缺点是关闭单个标签需要多一次点击，和本次“快速处理搜索结果”的目标不完全匹配。

推荐采用方案二。原因是本次只新增一个关闭动作，还不需要引入菜单系统；同时关闭入口不应常驻挤压搜索结果主体信息。

## 交互设计

搜索结果保留现有主行为：

- 点击结果主体中的“打开标签”区域：跳转到该标签页。
- `↑` / `↓`：移动选中结果。
- `Enter`：跳转到当前选中结果。

新增关闭行为：

- 当结果项对应已打开普通标签页时，鼠标悬停、结果项聚焦或键盘选中后显示关闭按钮。
- 点击关闭按钮只关闭该条结果对应的标签页。
- 关闭按钮点击时必须阻止结果项的默认跳转行为，原因是用户点击关闭时不应先切换到即将关闭的页面。
- 关闭成功后底部状态提示：`已关闭：页面标题`。
- 关闭失败后底部状态提示：`关闭失败，标签可能已经不存在`。

关闭按钮的可见文案可以使用符号或短文本，但无障碍名称必须使用中文：

```text
关闭标签：页面标题
```

如果页面标题为空，则使用网址或“未命名标签页”作为兜底名称。

搜索结果的 DOM 结构需要避免按钮嵌套。现有结果项根节点是按钮，新增关闭按钮时应调整为：

- 结果项根节点使用 `article` 或 `div`。
- 结果主体使用独立的“打开标签”按钮，承载原来的点击跳转行为。
- 关闭按钮和“打开标签”按钮作为兄弟节点存在。

这样设计的原因是 HTML 不允许按钮嵌套，兄弟按钮结构才能稳定处理点击、焦点和辅助技术语义。

样式迁移需要保持现有视觉行为：

- 结果项根节点继续使用 `.quick-result-item` 承载布局、悬停态和选中态。
- “打开标签”按钮使用新的主体按钮类承载标题、网址、分组和元信息。
- 关闭按钮使用独立类控制尺寸、显隐和危险操作视觉，不复用打开按钮样式。
- 结果项右侧需要保留固定宽度操作槽，关闭按钮默认弱化显示，悬停、聚焦或选中时增强显示。

这样拆分的原因是根节点负责整行状态，两个按钮分别负责不同动作，避免把整行点击样式和关闭按钮样式混在一起。固定操作槽可以避免关闭按钮显现时挤压标题和网址，减少搜索结果列表的横向抖动。

关闭按钮的视觉规格：

- 点击区使用固定方形尺寸，建议 `32px × 32px` 到 `40px × 40px`。
- 默认使用中性灰色和低强调背景，避免整个列表呈现过强危险感。
- 悬停、键盘聚焦或所在结果被选中时，再使用危险色强化可见性。
- 状态变化只使用 `opacity`、`background-color` 和 `border-color`，时长控制在 `80ms` 到 `120ms`。
- 不使用位移、宽高变化或阴影动画，原因是弹窗滚动列表需要避免布局抖动和重绘压力。

状态分层：

- 结果根节点负责整行悬停态和 `is-selected` 选中态。
- “打开标签”按钮负责主体点击区域、文本布局和标题省略。
- 关闭按钮负责自己的悬停态、聚焦环和危险操作反馈。

这种分层可以让用户分清当前高亮的是搜索结果，还是关闭按钮本身。

## 安全边界

以下结果不展示关闭按钮：

- 最近关闭结果：它们已经不是打开状态，关闭语义不成立。
- 固定标签：固定标签通常是用户长期保留入口，误关成本更高。
- 当前聚焦窗口的当前活动标签：用户正在使用该页面，误关会直接打断当前上下文。
- 正在播放音频的标签：关闭后可能中断用户正在收听或观看的内容。

这些限制只影响搜索结果内的快捷关闭入口，不改变浏览器本身的标签关闭能力，也不改变智能去重中的确认式关闭流程。

## 状态设计

前台状态继续以 `state.tabs` 作为已打开标签页快照来源。

关闭成功后：

1. 从 `state.tabs` 中移除对应 `tab.id`。
2. 保持当前搜索词不变。
3. 重新计算可见结果。
4. 如果当前选中索引越界，则收敛到最后一条结果。
5. 重新渲染搜索结果和可见结果数量。
6. 将焦点保持在结果列表上下文中，原因是用户刚刚在列表里执行关闭动作，关闭后通常还会继续处理相邻结果。

这里优先使用前台本地更新，原因是单个关闭动作结果明确，重新请求完整状态会造成弹窗闪动，也可能让键盘选中位置丢失。

当前窗口概览、重复提示和高级管理摘要可以在关闭成功后通过现有状态刷新兜底更新。第一阶段不要求这些摘要和本地搜索列表完全同步，原因是本次能力的主反馈对象是搜索结果列表本身。

如果后台关闭失败，前台不做本地移除，并展示失败提示。若失败原因可能来自状态过期，可以在失败后触发一次完整状态刷新作为兜底。

## 后台设计

后台新增一个消息动作，例如：

```js
{
  action: 'close-search-result-tab',
  payload: {
    tabId: 123
  }
}
```

后台处理逻辑：

1. 校验 `tabId` 必须是数字。
2. 调用 `chrome.tabs.get(tabId)` 获取最新标签页状态。
3. 若标签不存在，返回稳定中文错误。
4. 若标签是固定标签、当前聚焦窗口的当前活动标签或正在播放音频，则拒绝关闭。
5. 调用 `chrome.tabs.remove(tabId)` 关闭标签页。
6. 返回被关闭标签页的标题、网址和编号，供前台提示和本地移除。

后台重复校验安全边界，原因是前台状态可能过期，不能只依赖界面隐藏按钮来保证安全。

后台判断当前聚焦窗口的当前活动标签时，应结合浏览器当前聚焦窗口和目标标签 `windowId`。不能只依赖目标标签自身的 `active` 字段，原因是其他窗口的活动标签并不一定是用户当前正在看的页面，过度拦截会让跨窗口搜索结果无法被正常清理。

## 前台设计

前台在渲染搜索结果时判断是否可以关闭：

- `resultType` 必须是 `open` 或缺省值。
- `tab.id` 必须是数字。
- `tab.pinned` 不能为 `true`。
- 目标标签不能是当前聚焦窗口的当前活动标签，前台判断式应为 `tab.isCurrentWindow && tab.active`。
- `tab.audible` 不能为 `true`。

符合条件时，在结果项中渲染关闭按钮。按钮事件需要：

1. 调用 `event.stopPropagation()`，避免触发结果项跳转。
2. 仅在出现真实默认行为冲突时调用 `event.preventDefault()`，原因是普通按钮点击不需要默认阻止，过度阻止可能影响键盘和辅助技术行为。
3. 发送后台关闭消息。
4. 根据返回结果更新本地状态和界面。

结果项结构调整后，打开按钮继续复用原有结果主体样式和跳转事件；关闭按钮只处理关闭动作，不能绑定打开行为。关闭按钮只在当前结果项被用户关注时显示，样式可通过以下状态控制：

- `:hover`
- `:focus-within`
- 当前键盘选中项对应的类名

因为关闭后焦点会保持在结果列表上下文中，结果列表也需要支持现有搜索键盘行为：

- 当焦点位于结果列表容器或“打开标签”按钮时，`↑` / `↓` 继续移动 `state.selectedIndex`。
- 当焦点位于结果列表容器或“打开标签”按钮时，`Enter` 继续打开当前选中结果。
- 当焦点位于关闭按钮时，`Enter` 只触发关闭按钮自身点击，不额外触发打开结果。
- 搜索输入框保留现有键盘处理，结果列表键盘处理应复用同一段选择逻辑，避免两个入口的选中规则不一致。

这里需要拆出共享键盘处理函数，原因是现有 `handleSearchKeydown` 只绑定搜索输入框；如果关闭后焦点移动到结果按钮，继续依赖搜索输入框事件会导致方向键和回车失效。

## 可访问性

- 关闭按钮必须是原生 `button`。
- 关闭按钮必须有中文 `aria-label`。
- 键盘选中结果时，关闭按钮应进入可见状态，原因是键盘用户需要发现该操作。
- 不新增关闭快捷键，避免和搜索输入冲突。
- 结果列表容器需要设置 `tabindex="-1"`，作为列表为空时的焦点兜底。
- 关闭结果后，焦点应保持在结果列表上下文中。若关闭按钮所在结果被移除且列表仍有结果，则焦点移动到新的选中结果的“打开标签”按钮；若列表为空，则焦点停留在结果列表容器，不能跳到页面顶部或回到搜索输入框。

## 错误处理

- 标签已经不存在：提示 `关闭失败，标签可能已经不存在`。
- 标签不允许关闭：提示 `该标签受保护，未关闭`。
- Chrome API 调用失败：提示 `关闭失败，请稍后重试`。
- 关闭成功但本地状态更新后无结果：展示现有空状态文案。

错误提示保持短句，原因是弹窗空间有限，详细错误不应挤占搜索结果。

## 校验策略

本次继续扩展现有 `scripts/校验插件.js`，不新增测试框架。

需要覆盖：

- 后台存在关闭搜索结果标签的消息动作。
- 后台拒绝关闭固定标签、当前聚焦窗口的当前活动标签和正在播放音频的标签。
- 后台成功关闭普通标签并返回标题与编号。
- 弹窗结构中存在关闭按钮渲染逻辑。
- 搜索结果结构不出现按钮嵌套，打开按钮和关闭按钮必须是兄弟节点。
- 根节点、打开按钮和关闭按钮使用不同类名，避免把原有单按钮样式直接套到多个交互元素上。
- 右侧关闭操作槽宽度固定，关闭按钮显隐不得造成标题、网址或元信息横向跳动。
- 关闭按钮具备独立 hover 和 focus-visible 样式，默认状态不使用强危险色。
- 前台可关闭判断使用 `tab.isCurrentWindow && tab.active` 保护当前聚焦窗口的当前活动标签。
- 结果列表和搜索输入框复用同一套 `↑` / `↓` / `Enter` 选择逻辑。
- 关闭按钮事件处理逻辑包含阻止冒泡，降低误触结果项跳转的风险。
- README 增加搜索结果可关闭标签的中文说明。

现有校验脚本不运行真实浏览器交互，因此“点击关闭按钮不会触发结果项跳转”“关闭后焦点保持在结果列表”和“结果按钮获得焦点后方向键仍可切换选中项”需要作为人工验收项。脚本侧只检查事件处理函数和关键结构，避免用脆弱的字符串断言假装覆盖真实点击行为。

同时继续运行：

```bash
node scripts/校验插件.js
node --check background.js
node --check popup.js
```

## 分阶段实现

### 第一阶段：单个普通标签关闭

- 后台新增关闭消息。
- 前台搜索结果项新增关闭按钮。
- 关闭成功后本地移除结果。
- 补充校验脚本和 README。

### 后续可选阶段

- 增加复制链接、移动到当前窗口等搜索结果动作。
- 在更多操作菜单中承载多个动作。
- 根据用户反馈评估是否增加快捷键关闭。

这些能力暂不进入第一阶段，原因是本次目标是验证“搜索结果可直接处理标签”的最小闭环。

## 自检

- 本设计没有 `TODO`、`TBD` 或未定义占位内容。
- 本设计明确排除了批量关闭、快捷键关闭和最近关闭结果关闭。
- 本设计同时覆盖前台隐藏入口和后台强制校验，避免只靠界面状态保护高风险标签。
- 本设计明确关闭后焦点保持在结果列表，避免连续清理时被带回搜索输入框。
- 本设计明确搜索结果需要从单按钮结构调整为兄弟按钮结构，避免按钮嵌套。
- 本设计要求结果列表复用搜索输入框的键盘选择逻辑，避免焦点进入列表后方向键失效。
- 本设计明确关闭操作槽固定宽度和状态分层，避免按钮出现时造成列表抖动。
- 本设计没有修改一键整理、去重、工作集和分组规则的既有语义。
