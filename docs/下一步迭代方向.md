# 下一步迭代方向

## 当前判断

`标签之神` 的核心价值不是保存一套长期工作状态，而是在标签页过多时快速找到、切换、整理并减少混乱。工作集能力已经能覆盖少量恢复场景，但它不是高频入口，继续加导入、导出或详情管理容易把产品推向低频会话管理器，偏离当前插件最强的使用场景。

下一步应继续围绕高频标签处理迭代，优先提升用户每天打开弹窗时的效率和确定性。

## 优先方向

1. 搜索与切换体验增强

   搜索框是最高频入口。当前搜索已经支持标题、网址、主域名和分组名，但匹配方式仍偏直接包含。下一步优先增强多关键词、缩写和排序权重，让用户少打字也能命中目标标签。

2. 一键整理前预览

   标签数量很多时，用户会关心整理后哪些标签会移动、哪些分组会创建、哪些页面不会建组。预览可以提升整理动作的可信度，但会占用弹窗空间，应在搜索增强之后再设计。

3. 整理规则诊断

   自定义规则变多后，用户需要知道某个标签为什么进入某个分组。规则诊断适合服务高级用户，可以解释命中的规则、最终阈值和未创建原生分组的原因。

4. 重复标签处理增强

   现有确认式清理已经具备安全边界。后续可以增强重复原因说明、批量选择和保留策略，但优先级低于搜索与整理可信度。

## 暂不优先

- 工作集导入、导出和详情管理：使用场景较小，短期收益不足。
- 独立管理页面：可以解决弹窗空间问题，但只有当高级管理继续增长时才值得投入。
- 拼音搜索库：会引入依赖和体积变化，当前无构建链，先不作为近期方向。

## 第一阶段范围

第一阶段先做搜索与切换体验增强，目标是让用户在不改变现有入口和快捷键的情况下，更容易从大量标签里定位目标页面。

本阶段只做轻量增强：

- 支持空格分隔的多关键词搜索。
- 支持标题、分组名和网址片段的首字母缩写命中。
- 支持非连续字符的弱匹配，但分数低于精确和包含匹配，避免噪声结果压过明确结果。
- 保持现有键盘选择和 `Enter` 跳转流程不变。
